Below are the findings ordered by severity, with file/line refs.

  Findings (with victim mode)

  - Critical [mTLS]: TLS client does not perform hostname verification or set SNI, so any valid CA-signed cert
    for a different host will be accepted. gck-rpc-tls.c:389 only enables SSL_VERIFY_PEER without host
    checks, and gck-rpc-module.c:513 parses host but never passes it to OpenSSL. Add
    SSL_set_tlsext_host_name + X509_VERIFY_PARAM_set1_host (or set1_ip_asc for IP literals) on the
    client before SSL_connect.
    Status: Resolved (TLS peer host verification + SNI added; gck-rpc-tls.c and gck-rpc-module.c).
  - High [mTLS+PSK]: TLS enforcement can silently downgrade to plaintext if calloc for per-connection TLS state
    fails. gck-rpc-dispatch.c:2421 allocates tls without a NULL check and proceeds with tls == NULL, so
    a tls:// listener could accept non‑TLS connections. Fail the accept if TLS is required or propagate
    error.
    Status: Resolved (accept now fails if TLS state allocation fails).
  - High [mTLS+PSK]: TLS connection failure leaks resources and may leak the socket. In gck-rpc-module.c:530–570,
    errors return without freeing tls_ctx / cs->tls or closing sock. Then call_lookup frees cs directly
    (gck-rpc-module.c:659–662), bypassing call_destroy, so the leaks persist.
    Status: Resolved (cleanup path in call_connect + call_lookup uses call_destroy on failure).
  - Medium [PSK]: PSK client identity auto-detection uses the full line identity:hexkey as the identity. gck-
    rpc-tls.c:320–341 never splits at : so the server lookup won’t match unless the identity equals the
    entire line. Parse the identity part before : and trim whitespace.
    Status: Resolved (identity parsing now splits at ':' and trims).
  - Medium [mTLS+PSK]: gck_rpc_set_string truncates all config strings to 255 bytes regardless of the destination
    buffer size. gck-rpc-conf.c:74–78 makes long so_path, cert, or key paths silently invalid. Pass
    buffer size or use a size-aware helper per field.
    Status: Resolved (size-aware setter uses dest_len).
  - Medium [mTLS+PSK]: TLS read/write helpers treat SSL_ERROR_WANT_READ/WRITE as hard failures. gck-rpc-tls.c:517–
    542 and gck-rpc-tls.c:558–582 return -1 without retrying or signalling the caller to retry. This can cause sporadic failures under renegotiation or partial I/O.
    Status: Resolved (WANT_READ/WRITE retry loop in read/write helpers).
  - Low [mTLS+PSK]: parse_argument can dereference a NULL value when a key is specified without :/=. gck-rpc-
    module.c:124–142 pass value to snprintf even if it’s NULL. Guard against missing values or emit a
    warning.
    Status: Resolved (missing-value guard added).

  Open questions / assumptions

  - Are tls:// endpoints expected to always use mTLS with hostname verification? If not, do you want a
    config toggle for hostname checking or default‑on with opt‑out?
  - Should PKCS11_PROXY_CONF_PATH be treated as mandatory if set? Right now missing/unreadable config
    still returns success (gck-rpc-conf.c:173–185).

  Change summary

  - New TLS/mTLS support and config parsing are in place, but several error paths and verification steps are incomplete, and there are silent downgrade/cleanup risks that should be addressed before relying on TLS in production.

  Proposed fixes (all findings)

  - [mTLS] Add hostname verification + SNI on the client before SSL_connect. Use host from
    gck_rpc_parse_host_port(), detect IP literal vs DNS name, and set:
      - SSL_set_tlsext_host_name(ssl, host) for DNS names.
      - X509_VERIFY_PARAM_set1_host(SSL_get0_param(ssl), host) or
        X509_VERIFY_PARAM_set1_ip_asc(...) for IPs.
    Also consider a config flag for “skip hostname verification” if needed for lab setups.
  - [mTLS+PSK] Enforce TLS if tls_ctx is provided: check tls allocation in gck_rpc_layer_accept and fail
    the accept (close new_fd, log) if tls == NULL. This prevents accidental plaintext for tls://.
  - [mTLS+PSK] Fix TLS init failure cleanup in call_connect: on any error after socket creation, call
    gck_rpc_close_tls_all(cs->tls), free cs->tls->ctx, free cs->tls, and close(sock). Also ensure the
    call_lookup error path calls call_destroy (or a new cleanup helper) instead of free(cs).
  - [PSK] Parse identity from psk file line: split at ':' before setting tls_psk_identity and trim
    whitespace. Also skip blank/comment lines and handle lines without ':' gracefully.
  - [mTLS+PSK] Make config string setter size-aware: replace gck_rpc_set_string with a helper that takes
    dest buffer size; update each config option entry to pass size. This avoids silent truncation.
  - [mTLS+PSK] Handle SSL_ERROR_WANT_READ/WRITE with retry: either loop within
    gck_rpc_tls_read_all/write_all until completion or propagate a retryable status to callers (then
    have callers retry). Also consider SSL_MODE_AUTO_RETRY for blocking sockets.
  - [mTLS+PSK] Guard parse_argument against missing values: if value == NULL, log and skip or set
    defaults rather than calling snprintf with NULL.
